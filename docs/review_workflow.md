# Review workflow

## Цель

Review-процесс в `docustruct-ai` предназначен для selective human review: человек проверяет только спорные поля, а не весь документ целиком.

## Когда поле попадает в review

Поле отправляется в review, если:
- confidence ниже `accept_threshold`, но выше `reject_threshold`;
- отсутствует grounding;
- validation вернула ошибку или warning;
- документ низкого качества;
- есть расхождение между несколькими engines.

Опциональные пустые поля сами по себе не должны создавать review-шум. Если поле не является обязательным и корректно интерпретировано как отсутствующее, оно может быть auto-accepted без review task.

Если документ требует review по document-level причине, но ни одно отдельное поле не было отправлено в review, система создаёт document-level review task.

## Что видит reviewer

- исходную страницу;
- значение-кандидат;
- статус валидации;
- evidence text;
- `bbox`, если grounding найден;
- сырой OCR text;
- альтернативные кандидаты, если они были рассчитаны.

## Возможные действия

- `accept`
  Подтвердить извлечённое значение без правок.

- `edit`
  Исправить значение вручную и сохранить его как финальное.

- `unsupported`
  Явно отметить поле или документ как неподдерживаемый.

## Что сохраняется

В систему пишутся:
- review task;
- финальное решение;
- reviewer;
- timestamp;
- комментарий;
- финальное значение.

При ручном `edit` значение нормализуется по типу поля: даты приводятся к ISO-формату, числа парсятся как numeric values, строки очищаются от лишних пробелов. После завершения review document state пересчитывается повторно.

Эта структура пригодна для последующего:
- анализа качества;
- построения reviewer analytics;
- генерации datasets для active learning.
